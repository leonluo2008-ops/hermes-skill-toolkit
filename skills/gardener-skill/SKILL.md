# 园丁 skill

> 思维方式型 skill 需要的是「培育」，不是「优化」。
> 园丁不改造一棵树，而是让它长得更像它自己。
> **2026-06-01 扩展**：园丁也是「种树人」——写新 skill（§A）+ 养老 skill（§B）一站式。

---

## 何时用什么（路由总表）

**用户想做的事** → **用哪段**：

| 用户意图 | 触发词样例 | 对应章节 |
|---------|----------|---------|
| 写一个全新 skill | "写 skill"、"新建 skill"、"新做一个 skill" | **§A 写新 skill** |
| 改/优化现有 skill | "优化 skill"、"改 skill"、"skill 启发不好" | **§B 改老 skill**（原 gardener 主流程）|
| 写完想自检 | "skill 写完了"、"这样写对吗" | **§C 写完自检清单** |
| 不确定要写还是改 | "这个 skill 怎么搞" | **§D 路由决策树** |

**为什么合并**（2026-06-01 用户反馈）：
- 维护痛点：写 skill 之前要加载 authoring，改 skill 要加载 gardener——**两个 skill 维护累**
- 解决：园丁二合一，**一个 skill 解决一类事**（关于 skill 的全生命周期）

## §A 写新 skill 的视角（**轻量，主管道走 skill-creator**）

> **2026-06-01 调整**：**写新 skill 的主管道是官方 `skill-creator`**，不是园丁 §A。
> 本节只保留**园丁独有的视角**——三条铁律 + 落位。frontmatter / 目录结构 / evals 等"技术规范"已全部交给 skill-creator。

### A.1 三条不可违反的铁律（园丁视角）

1. **占位符思维 — Variable over hardcoded.** 任何会因机器/用户/场景而变的值，必须用 `{{VARIABLE}}` 或环境变量。三种情况：
   - 路径 → `__file__`-relative + env var override
   - ID/常量 → placeholder `{{CHAT_ID}}`，绝不 hardcode
   - 示例内容 → 提取**模式**，丢弃具体数据

2. **示例 ≠ 模板素材.** 用户展示的真实输出是参考材料——**不是要塞进 skill 的内容**。skill 编码的是**工作流骨架**，不是证词。

3. **原理 > 格式.** 写每节前问："这一步为什么存在？删了会破什么？"——**答案**才写进 skill。只编码格式（做什么）不编码原因（为什么）= 脆性 skill，不可迁移。

### A.2 SKILL.md 落位（园丁独有——官方没明确说）

两种位置：

1. **用户本地**：`~/.hermes/skills/<maybe-category>/<name>/SKILL.md` — 个人，不共享。用 `skill_manage(action='create')`。
2. **in-repo**（本 skill 主要是这种情况）：`/path/to/repo/skills/<category>/<name>/SKILL.md` — 提交随包发布。用 `write_file` + `git add`。`skill_manage(action='create')` **不**针对这个目录。

### A.3 主管道指引

**写新 skill 时直接走**：

| 步骤 | 工具 | 说明 |
|------|------|------|
| 1. 写 SKILL.md | **`skill-creator`** | 完整 frontmatter / 目录 / 描述触发词规范 |
| 2. 写 evals | **`skill-creator`** | `evals/evals.json` + test prompts |
| 3. 跑测试 | **`skill-creator`** | with-skill + baseline 对比 |
| 4. 评分 | **`skill-creator`** | 8 维评分 + viewer |
| 5. 优化 | `skill-creator` **或** `darwin-skill` | 思维类问题→园丁 / 流程类问题→达尔文 |

**A.4-A.5（原 frontmatter 规范 + 目录结构 + 自检）已删除**——**官方 skill-creator 全部覆盖**。

## §C 写完自检清单（写新 skill 后必跑）

**目的**：写完 SKILL.md 不直接 commit——先用这个清单自检。

| # | 检查项 | 通过标准 | 不通过怎么办 |
|---|--------|----------|------------|
| 1 | 三铁律覆盖 | 占位符思维、示例≠模板、原理>格式 三条都体现 | 缺的补到 SKILL.md |
| 2 | frontmatter 完整 | name / description / 触发词 都有 | 补 frontmatter |
| 3 | description ≤1024 字符 | 字符数检查 | 精简 |
| 4 | 触发词有中英文 | 至少 5 个，覆盖典型求助说法 | 加触发词 |
| 5 | 至少 1 个 examples 或反例 | 用户知道"做出来长啥样" | 加 example |
| 6 | 失败/边界情况 | "什么时候不该用"、"会卡在哪" | 加 pitfall |
| 7 | references 引用 | 引用的文件都存在 | 删引用 / 加文件 |
| 8 | 没有 hardcode 测试数据 | 任何具体值都是 `{{VAR}}` | 替换 |
| 9 | 没有 AI 套话 | 没有"作为一个 AI 助手..."、"当然可以..." | 删 |
| 10 | 角色 = 教练而非答案 | 告诉用户**怎么想**，不告诉**答案** | 重写"原则"段 |

**自检完后**：跑一次 "假装用户用这个 skill" 的对话，验证能 load、能产出、不卡死。

## 核心定位

| | 达尔文 | 园丁 |
|---|---|---|
| 目标 | 棘轮优化，分数只升不降 | 培育启发性，不追求分数 |
| 对象 | 操作流程型 skill | 思维方式型 skill |
| 评估 | 子 agent 对比打分（可量化） | 创作者 prompt 对比（人在回路） |
| 核心机制 | keep/revert 棘轮 | 启发质量偏好判断 |
| 触发模式 | 问题触发 | 问题触发 |

**触发时机**：用户使用 skill 执行任务时发现问题，或者 skill 调试开发时实测发现问题，主动触发园丁。

**被动原则**：没有用户指令，不主动扫描优化。

**例外**：在 Phase 5（测试验证）期间，园丁的被动原则**暂停**。此时园丁应主动监控对话质量，识别问题信号，在用户反馈之前主动介入。

---

## 思维方式型 skill 的识别

满足以下任一条件的 skill，应启用园丁模式（而非达尔文）：

1. **描述语言**：包含「思维方式」「创作方法」「启发者」「对话式」等关键词
2. **结构特征**：
   - 没有严格的线性步骤（没有「第1步→第2步→第3步」）
   - 包含「如果...呢」「想象」「看见」等启发性表达
   - 有「不是...而是...」的判断句
3. **内容特征**：
   - 有反面案例（错误做法）
   - 有边界说明（什么不是这个思维方式）
   - 有思维节点标注（如 [NODE: 发散节点]）
4. **自我声明**：skill 里明确说「不需要达尔文式优化」或「不要把它变成步骤清单」

## 架构设计：两层结构

园丁由两个核心部分构成：

```
┌─────────────────────────────────────────┐
│  底层优化部分                            │
│  · 诊断（Phase 0-1.5）                   │
│  · 建议生成（Phase 2）                   │
│  · 验证（Phase 5-5.5-6）                 │
│  · 循环优化（Phase 7）                   │
└─────────────────────────────────────────┘
                    ↓ 写入
┌─────────────────────────────────────────┐
│  skill画像库                            │
│  · 每skill一份画像                       │
│  · 每次优化的诊断+方案+验证结果           │
│  · 积累为「这类问题的解决方案」知识库     │
└─────────────────────────────────────────┘
```

**子agent职责边界**：

| Phase | 子agent职责 | 主agent职责 |
|---|---|---|
| Phase 1 | 分析对话，提取问题 | 协调子agent，监督分析质量 |
| Phase 1.5 | — | 主agent执行根本原因分析（这是判断，不是分析） |
| Phase 2 | 生成优化建议 | 审核建议，决定优先级 |
| Phase 3 | 设计测试场景和流程 | 审核测试设计 |
| Phase 5 | — | 监控对话，记录反应 |
| Phase 5.5 | — | 主agent执行成品扫描 |
| Phase 6 | — | 人类判断（用户） |

**子agent不可用时的fallback**：子agent不可用时，主agent承担所有任务，标注 `dry_run`。

---

**画像库数据结构**（每次优化记录）：

```
skill_name：[skill名]
时间：[优化时间]

【诊断】
问题：[具体描述]
根因：[一句话说明]

【优化操作】
改了什么：[具体改动]
为什么改：[对应诊断的哪条]

【验证】
结果：[用户反馈]
验证方式：dry_run / full_test
```

### skill画像库数据结构

每条优化记录包含：

```
skill_name：anna-russelmann-perspective
时间：2026-06-02

【诊断】
问题：成品草稿缺少世界观建立，冲突来得太突兀
根因：Anna第二阶段没有内置「建立世界」的前置步骤

【优化操作】
改了什么：第二阶段新增前三页世界观描写

【验证】
结果：用户确认「这个版本更好一点」
```

**画像库的作用**：
- 积累优化经验，新skill进来时匹配类似案例
- 可追溯：为什么这样优化？有记录可查
- 不是模板，是项目需要什么存什么

## 标准流程

```
Phase 0: 接收问题
Phase 1: 对话分析（收集问题）
Phase 1.5: 追问根本原因
Phase 2: 归类问题 + 生成建议
Phase 3: 设计测试场景
Phase 4: 等待确认
Phase 5: 人在回路测试
Phase 5.5: 成品扫描（主动检查草稿质量）← 新增
Phase 6: 人类判断
Phase 7: 循环优化
Phase 8: 提交 + 推送
Phase 9: 更新画像库
```

### Phase 0: 接收问题

用户触发：「园丁处理一下」

**来源**：
- 用户使用 skill 执行任务时发现问题
- skill 调试开发时实测发现的问题

**接收**：
1. 确认目标 skill 是什么（用户指定，或从对话上下文中推断）
2. 确认问题是什么（用户描述 / 实测中发现）
3. 确认这是在优化哪个 skill 的哪个方向

**如果用户只说「园丁处理一下」，没有指定skill**：
- 从对话上下文中推断最近使用过的 skill
- 主动问：「要优化哪个 skill？」

### Phase 0.5: 创建项目画像

**必须创建项目模块**，用于持久化存储项目内容。

```
项目目录结构：
├── {skill-name}-优化/           # 园丁优化项目
│   ├── README.md               # 优化目标 + 问题描述
│   ├── 对话分析.md              # 子agent从对话中提取的问题点
│   ├── 优化建议.md              # 基于分析生成的优化方向
│   ├── 测试设计.md              # 测试场景设计
│   └── 验证记录.md              # 测试验证结果
│
└── {创作项目}/                  # 如果有创作项目（如森林小镇）
    ├── README.md               # 项目概述
    ├── 决策记录.md              # 关键决策点
    └── 当前进度.md              # 阶段状态
```

**创建后展示给用户确认**：
> 「已创建项目画像：[目录结构]。确认继续？还是需要调整？」

**保存时机**：
- Phase 完成后
- 关键决策点（用户纠正、选择方案等）
- 用户说「停」「不对」「重新想」时

**保存格式**：Markdown，AI易读

**内容由项目决定**：不是固定模板，是项目需要什么存什么

### Phase 1: 对话分析（子 agent）

子 agent 专门分析对话记录，提取问题点。

**子 agent 任务**：

1. **回顾对话**：
   - 分析从 Phase 0 到当前的完整对话
   - 识别用户反复纠正、调整的地方
   - 识别「说到了但没完成」的地方

2. **提取问题**：
   - 什么是用户反复说要改的？
   - 什么是用户在对话中纠正过的？
   - 什么是对话暴露的设计缺陷？

3. **更新项目画像**：
   - 写入「对话分析.md」
   - 标记「关键纠正点」

**关键原则**：分析是为了提取「需要优化的地方」，不是总结对话内容。

---

### Phase 1.5: 追问根本原因

> **只运行于 Phase 1 之后。** 这是区分「修流程」和「改定位」的关键步骤。

对 Phase 1 提取的每个问题，追问一层：

**判断标准**：

| 问题类型 | 判断方法 | 解决方法 |
|---|---|---|
| **流程缺失** | 通过「增加环节/步骤」能解决吗？ | 在工作流里补步骤 |
| **能力不足** | 现有角色定位无法支撑这个需求吗？ | 在 skill 定位里改角色 |
| **方法不当** | 以上都不是？ | 建议换方向 |

**判断决策树**：

```
问题：{问题描述}
    ↓
增加一个步骤/环节，能解决这个问题吗？
    ↓ 能解决 → 「流程缺失」
    ↓ 不能解决 → 继续
        ↓
这个问题的原因，是 skill 的「角色定位」不够吗？
    ↓ 是 → 「能力不足」→ 建议改 skill 定位
    ↓ 否 → 「方法不当」→ 建议换方向
```

**判断标准详解**：

- **流程缺失**：问题出在「缺少一步」，加了就能解决
  - 信号：输出没有格式规范 → 加输出模板
  - 信号：没有用户确认点 → 加 Phase 4
  - 信号：成品草稿没人检查 → 加 Phase 5.5

- **能力不足**：问题出在「角色不够」，加步骤也解决不了
  - 信号：「Anna 不适合整部系列创作」（单集→整部，角色变了）
  - 信号：「用户想要AI输出成品，Anna 设计是用户输出」（定位冲突）
  - 信号：加了约束/步骤后问题依然存在

- **方法不当**：问题出在「方向/方法」本身
  - 信号：用错了创作框架
  - 信号：参考作品分析方向有误

**输出**：

每个问题输出：
```
问题：[问题描述]
类型：流程缺失 / 能力不足 / 方法不当
根因：[一句话说明]
建议方向：[描述]
```

**更新项目画像**：写入「对话分析.md」的「根本原因分析」章节

---

### Phase 2: 建议生成（子 agent）

基于 Phase 1 的分析，生成优化建议。

**子 agent 任务**：

1. **归类问题**：
   - 这个问题属于哪种优化类型？（P1-P5）
   - 根因是什么？（设计缺陷？场景不匹配？方法不对？）

2. **生成建议**：
   - 具体的优化方向
   - 每个方向的预期效果
   - 优先级排序

3. **更新项目画像**：
   - 写入「优化建议.md」
   - 每个建议说明「解决了什么问题」

---

### Phase 3: 设计测试（子 agent）

设计测试场景。

**子 agent 任务**：

1. **设计测试场景**：
   - 基于优化方向，设计能触发改动的测试 prompt
   - 确保测试场景真实（用户真正想写的题材）
   - 确保能触发「有改动的那个部分」

2. **设计测试流程**：
   - 人 + AI 一起对话的步骤
   - AI 观察记录的内容
   - 判断标准

3. **更新项目画像**：
   - 写入「测试设计.md」

---

### Phase 4: 等待确认

展示分析结果和优化建议，等用户确认：

**展示内容**：
- Phase 1 的问题分析汇总
- Phase 2 的优化建议方向（1-2个选项）
- 每个方向的预期效果

**用户决策**：
- 确认方向 → 进入 Phase 5
- 否定/修改 → 回到 Phase 1 重新分析
- 「不好」→「停」→「重新想」→ 回到 Phase 2

**Phase 5.5 成品扫描说明**：
- 如果 Anna 已输出成品草稿，Phase 4 前先执行 Phase 5.5
- Phase 5.5 发现问题 → 先问用户要不要改 → 用户确认后再进入 Phase 5 测试

---

### Phase 5: 测试验证

**思维类 skill 的测试，必须有人参与，不是 AI 跑对比。**

---

#### 为什么不能只靠 AI 测试

- AI 没有真正的「创作灵感」
- AI 的感受不是人的感受
- 人类判断的是「这个 skill 能不能启发我」，不是「AI 跑出来的输出」
- 模板化的输出可能是 AI 没有发现更好的表达方式，不是 skill 的问题

---

#### 正确的测试方式

**人 + AI 一起执行测试验证**：

```
AI 发起对话
    ↓
人作为创作者参与对话
    ↓
人在对话过程中实时感受到启发性
    ↓
AI 观察并记录人的真实反应
    ↓
对话结束后，AI 总结人的反馈
    ↓
判断：优化方向是否有效
```

---

#### 关键原则

1. **人必须真实参与对话**，不是看 AI 整理的输出
2. **AI 是观察者**，记录人在对话中的反应（犹豫/兴奋/困惑/选选项）
3. **关注对话过程**，不是关注输出结果
4. **测试场景要真实**：用用户真正想写的题材，不是随意编一个 prompt

---

#### AI 监控探索阶段和执行阶段的边界

思维类 skill 的对话分为两个阶段：
- **探索阶段**（定框架）：定故事方向、定核心角色、定世界设定
- **执行阶段**（写作）：具体写故事、文字输出

AI 在测试中要监控：
- 创作者是否跳到了「执行阶段」的思维（问「要教什么」「故事告诉我们什么」）
- 对话是否陷入了「清单式回答」而不是「探索式思考」
- 是否有「过早收拢」的现象（还没探索完就想要结论）

**发现问题时的介入原则**：

当创作者出现以下情况时，园丁要打断并拉回探索阶段：
- 问「这个故事能教小朋友什么」→ 拉回到「森林里发生了什么意外」
- 直接说「我要写一个关于XX的故事，目标是XX」→ 拉回到「为什么你想写这个」
- 跳过探索直接想要结论 → 拉回到「你脑子里现在有什么画面」

介入方式：用提问打断，不是给答案
- 「停，你先告诉我，这个故事里发生了什么意外？」
- 「在你说教育目标之前，我们先想清楚，森林里出了什么事？」

#### 什么情况下测试无效

- AI 在对话中机械执行节点，而不是真正对话

### Phase 5.5: 成品扫描（主动检查）

**目的**：在 Anna 输出成品草稿后，园丁主动检查草稿质量，不等用户发现。

**什么时候执行**：
- Anna 完成第二阶段（输出完整剧集）后
- 在用户反馈之前
- 自动执行，不需要用户触发

**检查维度**：

| 维度 | 检查什么 | 问题信号 |
|---|---|---|
| 世界感 | 故事开头是否建立了世界和角色？ | 一来就是矛盾事件，太突兀 |
| 节奏 | 冲突是否来得太突兀？有没有铺垫？ | 用户说「太突然」 |
| 意外+成长 | 两者是否绑定？ | 只有意外没有成长 |
| 目标词分布 | 是否自然分布？ | 生硬植入、重复堆砌 |

**发现问题时的操作**：
1. 扫描发现维度有问题
2. **先问用户，不直接修改**
   > 「我注意到[维度]有问题：[具体描述]。要现在修改吗？」
3. 用户确认后 → 进入第三阶段（迭代优化）

**注意**：扫描是主动检查，发现问题后**问用户要不要改**，用户决定后再执行修改。

### Phase 6: 人类判断

展示分析结果和优化建议，问用户：

```
基于对话分析，发现以下问题：
[问题1]
[问题2]
...

建议的优化方向：
[方向1]
[方向2]
...

你确认哪个方向？
```

**判断标准（人在回路）**：

| 评估点 | 说明 |
|---|---|
| 创作者能形成自己的画面吗？ | skill 给了启发而不是答案 |
| 有没有套路感？ | 还是说有新意 |
| 开放性有没有被压缩？ | 还是说被「步骤化」了 |
| 整体启发性变好了吗？ | 偏好 A 还是 B |

**决策**：
- 确认优化方向 → 进入 Phase 7
- 否定/修改 → 回到 Phase 1 重新分析
- 不满意 → 继续迭代（回到 Phase 2）

## 设计原则

1. **被动接收**：没有用户指令，不主动扫描优化
2. **固定方法**：提取方法（收集→归类→验证→沉淀）是标准步骤，不是可选工具
3. **项目画像**：每个优化任务独立项目模块，阶段性成果实时保存
4. **子 agent 架构**：Phase 1/2/3 由子agent执行，Phase 1.5/5.5/6 由主agent/用户执行（详见职责边界表）
5. **人在回路**：所有判断由人类完成，园丁只提供信息和选项
6. **可回滚**：所有改动在分支上，人类判断前不修改 dev/master
7. **版本可见**：所有分支推送到 GitHub，用户可以查看 diff
8. **循环优化**：不满意就继续迭代，直到用户满意

## 禁止的优化方向（任何版本都必须拦截）

| 禁止 | 原因 |
|---|---|
| 增加「第X步」数量 | 把启发变成执行，压缩创作空间 |

## 实测发现的 pitfall（2026-06-02）

### Pitfall: 园丁判断深度不足——停在「症状修复」而非「根因定位」

**触发条件**：第一轮优化时，问题暴露为「Anna 不适合整部系列创作」，园丁的分析停留在加节点零前置约束，没有追问「 Anna 的定位是否需要改变」。

**错误后果**：
- 第一轮做了「修补」而非「重构」
- 问题只解决了一半（加了约束，但 Anna 的角色定位没改）
- 第三轮用户主动提出「 Anna 要成为整部作品创作 skill」，说明第一轮没有触及根本

**正确做法**：Phase 1.5（追问根本原因）是必须的——对每个问题多问一层：「这是流程缺失还是能力不足？」如果判断是「能力不足」，要建议改 skill 定位，不能只做修补。

**本次教训**：第一轮只做了 `f772957`（加约束），第三轮才发现需要 `3145908`（重构定位）。中间多走了一轮，浪费了沟通成本。

**检测方法**：当问题涉及「某个 skill 整体不适合某个场景」时，优先追问「定位」而非「流程」。

### Pitfall: Anna 在没有上下文时被直接调用节点一

**触发条件**：创作者只有模糊想法（「松鼠+森林小镇」几个字），没有任何系列/框架概念时，Anna 被直接调用并进入节点一（单集提问）。

**错误后果**：
- 创作者无法接住「第一集第一个意外是什么」这类问题
- 对话陷入「AI给选项 → 创作者说都行 → 继续尬聊」的死循环
- 绕了一大圈才发现应该先定框架

**正确做法**：Anna 启动时先判断「系列」还是「单集」——如果是系列任务，先建立节点零（系列框架），不跳到节点一。

**检测方法**：当创作者说「没细想法」「你来设计」时，说明缺乏上下文，应回到节点零先定框架。

**关联 skill**：已修复 `anna-russelmann-perspective`，增加了「适用场景判断」章节 + 节点零前置约束。

### Pitfall: 用户偏好信号未被感知——「先看全貌再决定」

**触发条件**：Anna 按标准流程推进（世界→角色→系列主线→目录→单集），但用户说「先看规划，再决定」。

**错误后果**：
- 用户想看到26集完整目录后再决定是否继续
- Anna 按流程推进到了「第一集故事设计」，跳过了用户的决策节点
- 用户的创作节奏偏好没有被 Anna 感知

**正确做法**：在第二层（目录规划）开始时，主动问：「你想先看完整的26集目录，还是边走边定？」

**检测方法**：当用户说「先看规划」时，说明用户有「先看全貌再决定」的偏好，Anna 需要调整节奏，不能按固定流程推进。

**本次教训**：Anna 三层结构（系列→目录→单集）是线性推进，但用户可能想要在目录阶段先看到全貌。这是 Anna 缺乏「感知用户创作节奏偏好」的能力。

### Pitfall: 目录展示格式影响用户判断

**触发条件**：Anna 给出「按字母族分组」的目录（A/T/I/N/S一组，O/C/K/E/U一组……），用户说「不行，你这种分类不方便记忆」，要求「肯定按字母表啊」。

**错误后果**：
- 目录展示方式影响用户对项目本身的判断
- 用户可能因为展示方式否定内容
- 说明 Anna 给方向时没有先确认「你想要什么样的展示方式」

**正确做法**：在目录规划开始时，先问：「目录想怎么展示？按字母表顺序？还是按字母族分组？」

**检测方法**：当用户纠正展示方式时，说明 Anna 跳过了「确认展示偏好」这一步。

**本次教训**：「按字母表顺序」是用户的默认偏好，Anna 在做目录规划时应先问展示方式，而不是给出自己认为合理的分组。

### Pitfall: 园丁需要「主动监控」机制，不能只靠用户触发

**触发条件**：测试 Anna 时，园丁处于「等待用户触发」状态，而不是「监控对话质量」状态。

**错误后果**：
- 问题暴露了，但园丁没有主动识别
- 用户说「我发现问题了」，园丁才介入
- 失去了在对话过程中及时发现问题的机会

**正确做法**：Phase 5.5（成品扫描）是主动监控机制的核心——在 Anna 输出成品草稿后，园丁主动检查四个维度：世界感、节奏、意外+成长绑定、目标词分布。发现问题主动指出。

**本次教训**：Phase 5.5（成品扫描）是必须的——问题被用户发现而非园丁主动识别，说明扫描机制缺失。

### Pitfall: Phase 5.5 已定义但未触发——扫描机制需要显式调用

**触发条件**：Anna 输出第一版成品草稿后，园丁没有执行 Phase 5.5 主动扫描，而是等待用户反馈。

**错误后果**：
- 用户先发现「缺少世界观，太突兀」
- 园丁失去了主动识别问题的时机
- 扫描机制存在，但没有被调用

**根因**：园丁没有把「Anna 输出成品草稿」识别为 Phase 5.5 的触发信号。

**正确做法**：
- Anna 完成第二阶段（输出完整剧集）后，园丁自动进入 Phase 5.5
- 不需要用户触发，Anna 输出成品 → 园丁立即扫描
- 扫描完成后才进入等待用户反馈状态

**检测方法**：当 Anna 输出的是完整剧集（不是故事种子）时，园丁应该识别为 Phase 5.5 触发信号。

**本次教训**：扫描机制需要和 Anna 的输出状态绑定——Anna 输出完整剧集 = 触发扫描，不需要用户指令。

### Pitfall: 用达尔文检测园丁时，先全面分析工作方式，再设计测试prompt

**触发条件**：用户说「用达尔文检测园丁」，直接按达尔文的「典型prompt」设计测试，而不是先理解园丁的实际工作方式。

**错误后果**：
- 设计的测试prompt不符合园丁的真实使用场景
- 检测结果偏离实际

**正确做法**：
1. **先分析园丁的工作方式**：读 SKILL.md，理解园丁的定位、流程、核心机制
2. **再设计测试prompt**：基于理解，设计符合园丁实际工作的测试场景
3. **最后进入评估**：用设计好的测试prompt跑基线评估

**本次教训**：达尔文的 Phase 0.5（测试prompt设计）不是套模板——需要先理解skill，再设计prompt。直接套模板会导致「测了但没测到点子上」。

### Pitfall: 不要因内容重叠就立刻合并 skill——先问用户的"维护痛点"

**触发条件**：发现两个 skill 在内容上有重叠区（如 gardener-skill 和 hermes-agent-skill-authoring 都有"skill 怎么写得好"的原则），立刻提议合并。

**错误后果**：
- 没问到用户真正的痛点（**是"内容重复"还是"维护累"**）
- 合并后反而丢了 skill 的边界清晰
- 用户没参与决策，方向走偏

**正确做法**：
1. **先问用户的痛点是哪个**：
   - "内容重复" → 抽公共章节到 references
   - "维护累" → 合并成一个 skill（**用内部章节区分使用时机**）
   - "找不到" → 加 related_skills 引用
2. **不同痛点对应不同操作**，不要默认"合并"
3. 合并时**用 "何时用什么" 路由表**做内部分区，不靠分类目录

**本园丁 pitfall 的教训**（2026-06-01）：
- 第一次分析 gardener vs hermes-agent-skill-authoring 时，倾向"不合并、保持边界"
- 用户纠正："**我维护的痛点：写一个 skill 要加载 2 个 skill，累。**"
- 正确解法：**合并 → 内部用 §A/§B/§C/§D 路由**
- 教训：**不要把"清晰边界"当成最高原则**——用户维护成本是更高的原则

**检测方法**：当发现两个 skill 重叠时，先问"你**维护**时有什么痛点"——用户的回答决定合并还是引用。

### Pitfall: §A 写新 skill 时偷懒——只写 frontmatter 不写正文

**触发条件**：用户说"我写个新 skill"，园丁只检查 frontmatter 完整就放行。

**错误后果**：
- SKILL.md 触发词齐全但**正文是模板化的空话**
- 用户跑起来发现"按它说的做了还是不会"
- 园丁失去了 §A 阶段的质量把关

**正确做法**：
- §A 写完必须**跑 §C 自检清单**（10 项）
- 第 5 项（至少 1 个 examples）和第 10 项（角色 = 教练）是**必查项**
- 自检不通过 → 改完再放行

**检测方法**：SKILL.md 长度 < 2KB 或没有 examples → §A 没做完。

### Pitfall: 园丁自己被"4-skill 工具包"诱惑——以为园丁要管"编写"

**触发条件**：发现官方 `skill-creator` 后，园丁想"它跟我分工重复"—— 想"自己管编写，skill-creator 只管 evals"。

**错误后果**：
- **官方 skill-creator 能力远超"编写"**（创建+编辑+优化 description+evals）
- 园丁本来是"优化"，**去抢"编写" = 越界**
- 用户已经明确分工："**用官方的 skill-creator 专门负责编写 skill，园丁和达尔文是优化**"
- 园丁的"种树人"角色（§A）**跟 skill-creator 撞车**——**该让位就让位**

**正确做法**：
- **尊重用户分工**（不是按"功能完整度"判断谁该管什么）
- 园丁保留**优化 + 思维诊断**这一定位
- "编写"交给 skill-creator，**§A 重构或删除**（用户拍板）
- 触发词/decision tree 里**写清楚"编写找 skill-creator，优化找园丁"**

**本园丁 pitfall 教训**（2026-06-01）：
- 第一版描述"园丁也是种树人——写新 skill (§A) + 养老 skill (§B) 一站式"——**越界了**
- 官方 skill-creator 出来后，**§A 该让位**
- 教训：**用户的分工意图比"功能完整度"更重要**

**检测方法**：当园丁想"扩展自己的职责"时，**先问用户"这个职责归谁管"**，不要默认"我管得越多越好"。

### Pitfall: 园丁"自作主张"——"边界"是用户定的不是园丁定的

**触发条件**：园丁发现两个 skill 似乎内容重叠（如 gardener + hermes-agent-skill-authoring），立刻提议"整合"或"保持独立"，**不问用户真正的痛点**。

**错误后果**（2026-06-01 实测）：
- 园丁先提议"保持独立"——错了（用户痛点是"维护累"）
- 园丁又提议"合并成一个"——错了（用户想用官方的 scale）
- 园丁再提议"删除 hermes-agent-skill-authoring"——错了（用户根本没想到合并，是想换官方）
- **3 次提议都错**——**每次都基于园丁自己的判断而非用户的"明确分工"**

**正确做法**：
- **不要预设答案**——发现 skill 看起来"重叠"时，**先问用户"你维护时有什么痛点"**
- **用户的"分工"决定方案**——内容重复 vs 维护累 vs 找不到 = 不同方案
- **园丁是"诊断工具"不是"决策工具"**——**边界由用户定**

**本 pitfall 教训**（2026-06-01）：
- 用户原话："**我打算使用官方的 skill 原 scale 来编写，园丁肯定是负责优化的，你等我通知**"
- 教训：**用户的"明确分工"优先级 > 园丁的"内容分析"**
- 检测：园丁开始提"整合/合并/拆分"时，**先停下来问用户**

### Pitfall: 用户说"等通知"——园丁就停，不要继续设计

**触发条件**：用户明确说"你等我通知"或"等我说再做"。

**错误后果**：
- 园丁继续自作主张设计合并方案、写 reference、commit、push
- 用户回来发现一堆操作记录——**破坏信任**
- 用户的"等通知" = **明确停止信号**

**正确做法**：
- 收到"等通知"信号，**立即停止所有动作**
- 只**做最少的信息整理**（如：读完 4 个 skill、列差异、待用户拍板）
- **不 commit、不 push、不修改任何文件**
- 等用户回来再行动

**本 pitfall 教训**（2026-06-01）：
- 用户说"我打算使用官方的 skill 原 scale 来编写，来负责编写 skill，园丁肯定是负责优化的，你等我通知"
- 园丁已经准备好 4 步合并计划——**应该停**
- 检测：**收到"等通知"字样时，园丁的优化流程暂停**

### Pitfall: 修改 skill 仓库前必做 git 同步检查——本地 dev 跟 origin 不一致会丢别人的 commit

**触发条件**：修改一个独立 git 仓库的 skill（`~/.hermes/skills/<skill>/` 是独立 repo）然后 commit，**不**先 fetch + 检查 origin 跟本地 HEAD 是否一致。

**错误后果**（2026-06-01 darwin-skill 实测）：
- 本地 dev HEAD = `2056abf`（指向 master 内容）
- origin/dev HEAD = `c6ac79f`（有 143 行"思维方式型skill保护模式"）
- **没察觉 diverged**——在本地改了 7 行 frontmatter 直接 commit
- diff 显示 `-143 / +41` = **静默丢了别人的 143 行**
- 立即 reset + reset --hard origin/dev 才恢复
- 重新 patch +7 行 → 干净 commit

**正确做法**（**所有 skill 仓库 commit 前必跑**）：
```bash
# 1. 拉最新
git fetch origin <branch>

# 2. 验证一致性
git rev-parse HEAD               # 本地
git rev-parse origin/<branch>    # 远程
# 必须**完全一致**才继续

# 3. 不一致时：reset --hard 同步（**别 rebase 别人的 commit**）
git reset --hard origin/<branch>
# 重新 patch 你的改动

# 4. 验证 diff 范围
git diff --stat
# **预期**：改动量是预期的一半以内
# 超过 10 倍 = 立即 git reset + 重新做
```

**判断标准**：
- `git diff --stat` 改动量 **≤ 预期** → 安全
- 改动量 **= 预期 × 2-5** → 检查是不是合并了别人的内容
- 改动量 **> 预期 × 10** → 立即 `git reset --hard origin/<branch>` + 重新做

**预防配置**（强烈建议每个 skill 仓库加）：
```bash
# pre-commit hook（hooks/pre-commit）
#!/bin/bash
EXPECTED_LINES=20  # 每次改的预期行数
ACTUAL=$(git diff --cached --stat | tail -1 | grep -oP '\d+' | head -1)
if [ "$ACTUAL" -gt $((EXPECTED_LINES * 10)) ]; then
  echo "ERROR: diff 改动量 $ACTUAL 超过预期 $EXPECTED_LINES × 10"
  echo "可能本地 HEAD 跟 origin 不一致，先 git fetch + reset --hard origin/<branch>"
  exit 1
fi
```

**关联**：完整 git 同步 + 安装流程 + push 网络 workaround 见 `hermes-multi-agent/references/skill-installation-procedure.md`。

### Pitfall: skill 库管理（整理/审计/归档）不归园丁管——归 skill-organizer

**触发条件**：用户说"skill 太多"、"哪些该删"、"整理 skill 库"——**园丁想接管**。

**错误后果**：
- 园丁的"项目画像"机制是**单 skill 视角**——**处理不了"全 skill 集合"**
- skill-organizer 的"5 步流程 + 双轨分类"是**专门为全 skill 集合设计**
- 园丁去管 = **职责越界 + 工具不对**

**正确做法**：
- 全 skill 集合问题（整理/审计/归档/分类）→ 转发给 **skill-organizer**
- 单 skill 优化（思维/启发质量）→ 园丁自己管
- 触发词清晰区分：园丁触发 = "优化/诊断（单 skill）"；skill-organizer 触发 = "整理/审计/分类（多 skill）"

**检测方法**：用户提到"全局"或"多个 skill"时，**先考虑 skill-organizer**，不要默认"我管"。
