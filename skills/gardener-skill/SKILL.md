# 园丁 skill

> 园丁读对话——从你和 skill 的真实交互里，挖出这个 skill 哪里不对、应该怎么改。
> **核心定位（2026-06-02 调整）**：园丁是「**对话信号驱动的 skill 优化工具**」。
> 不是「优化思维方式」，是**读对话 + 出方案**。输入是真实对话记录，输出是针对某个 skill 的具体优化建议。
> **2026-06-01 扩展**：园丁也是「种树人」——写新 skill（§A）+ 养老 skill（§B）一站式。

---

## 何时用什么（路由总表）

**用户想做的事** → **用哪段**：

| 用户意图 | 触发词样例 | 对应章节 |
|---------|----------|---------|
| 写一个全新 skill | "写 skill"、"新建 skill"、"新做一个 skill" | **§A 写新 skill** |
| 改/优化现有 skill | "优化 skill"、"改 skill"、"对话里这个 skill 不对" | **§B 改老 skill**（原 gardener 主流程）|
| 写完想自检 | "skill 写完了"、"这样写对吗" | **§C 写完自检清单** |
| 不确定要写还是改 | "这个 skill 怎么搞" | **§D 路由决策树** |
| 园丁方案出完，要量化验证 | "evals 跑分"、"量化验证"、"打分验证" | **Phase 9.5 → 达尔文**（见"园丁 → 达尔文 交接协议"）|

**为什么合并**（2026-06-01 用户反馈）：
- 维护痛点：写 skill 之前要加载 authoring，改 skill 要加载 gardener——**两个 skill 维护累**
- 解决：园丁二合一，**一个 skill 解决一类事**（关于 skill 的全生命周期）

## §A 写新 skill 的视角（**轻量，主管道走 skill-creator**）

> **2026-06-01 调整**：**写新 skill 的主管道是官方 `skill-creator`**，不是园丁 §A。
> 本节只保留**园丁独有的视角**——三条铁律 + 落位。frontmatter / 目录结构 / evals 等"技术规范"已全部交给 skill-creator。

### A.1 三条不可违反的铁律（园丁视角）

1. **占位符思维 — Variable over hardcoded.** 任何会因机器/用户/场景而变的值，必须用 `{{VARIABLE}}` 或环境变量。三种情况：
   - 路径 → `__file__`-relative + env var override
   - ID/常量 → placeholder `{{CHAT_ID}}`，绝不 hardcode
   - 示例内容 → 提取**模式**，丢弃具体数据

2. **示例 ≠ 模板素材.** 用户展示的真实输出是参考材料——**不是要塞进 skill 的内容**。skill 编码的是**工作流骨架**，不是证词。

3. **原理 > 格式.** 写每节前问："这一步为什么存在？删了会破什么？"——**答案**才写进 skill。只编码格式（做什么）不编码原因（为什么）= 脆性 skill，不可迁移。

### A.2 SKILL.md 落位（园丁独有——官方没明确说）

两种位置：

1. **用户本地**：`~/.hermes/skills/<maybe-category>/<name>/SKILL.md` — 个人，不共享。用 `skill_manage(action='create')`。
2. **in-repo**（本 skill 主要是这种情况）：`/path/to/repo/skills/<category>/<name>/SKILL.md` — 提交随包发布。用 `write_file` + `git add`。`skill_manage(action='create')` **不**针对这个目录。

### A.3 主管道指引

**写新 skill 时直接走**：

| 步骤 | 工具 | 说明 |
|------|------|------|
| 1. 写 SKILL.md | **`skill-creator`** | 完整 frontmatter / 目录 / 描述触发词规范 |
| 2. 写 evals | **`skill-creator`** | `evals/evals.json` + test prompts |
| 3. 跑测试 | **`skill-creator`** | with-skill + baseline 对比 |
| 4. 评分 | **`skill-creator`** | 8 维评分 + viewer |
| 5. 优化 | `skill-creator` **或** `darwin-skill` 或 `gardener-skill` | evals 分数下降→darwin / 对话暴露问题→园丁 / description→skill-creator |

**A.4-A.5（原 frontmatter 规范 + 目录结构 + 自检）已删除**——**官方 skill-creator 全部覆盖**。

## §C 写完自检清单（写新 skill 后必跑）

**目的**：写完 SKILL.md 不直接 commit——先用这个清单自检。

| # | 检查项 | 通过标准 | 不通过怎么办 |
|---|--------|----------|------------|
| 1 | 三铁律覆盖 | 占位符思维、示例≠模板、原理>格式 三条都体现 | 缺的补到 SKILL.md |
| 2 | frontmatter 完整 | name / description / 触发词 都有 | 补 frontmatter |
| 3 | description ≤1024 字符 | 字符数检查 | 精简 |
| 4 | 触发词有中英文 | 至少 5 个，覆盖典型求助说法 | 加触发词 |
| 5 | 至少 1 个 examples 或反例 | 用户知道"做出来长啥样" | 加 example |
| 6 | 失败/边界情况 | "什么时候不该用"、"会卡在哪" | 加 pitfall |
| 7 | references 引用 | 引用的文件都存在 | 删引用 / 加文件 |
| 8 | 没有 hardcode 测试数据 | 任何具体值都是 `{{VAR}}` | 替换 |
| 9 | 没有 AI 套话 | 没有"作为一个 AI 助手..."、"当然可以..." | 删 |
| 10 | 角色 = 教练而非答案 | 告诉用户**怎么想**，不告诉**答案** | 重写"原则"段 |

**自检完后**：跑一次 "假装用户用这个 skill" 的对话，验证能 load、能产出、不卡死。

## §D 路由决策树（不确定要写还是改时用）

```
用户说：[用户原话]
    ↓
【Q1】是否在描述"全局 skill 集合"？（"skill 太多"/"整理"/"哪些该删"）
    ↓ 是 → skill-organizer
    ↓ 否 → 继续
        ↓
【Q2】是否在描述"现有 skill 的问题"？
    ↓ 是 → §B 改老 skill（对话暴露问题→园丁 / evals 分数下降→darwin-skill）
    ↓ 否 → 继续
        ↓
【Q3】是否在描述"我想要一个新能力/新 skill"？
    ↓ 是 → skill-creator（**主管道**）
    ↓ 否 → 继续
        ↓
【Q4】是否在描述"我想要优化某个 skill"？
    ↓ 是 → 对话里发现的问题？→ §B 园丁
```

- "skill 写完了你帮我看看" → `skill-creator`（走 evals）
- "skill 太多了" → `skill-organizer`
- "对话里这个 skill 跑出来不对" → §B 园丁
- "evals 分数掉了" → `darwin-skill`
- "我之前写的那个 skill 优化下 description" → `skill-creator`

## 核心定位

| | 达尔文 | 园丁 |
|---|---|---|
| 目标 | 棘轮优化，分数只升不降 | 从对话里挖问题，给出优化方案 |
| 输入 | evals 量化测试 | 真实对话记录 |
| 对象 | 不限（量化评分通用） | 不限（对话信号通用） |
| 评估 | 子 agent 对比打分（可量化） | 创作者 prompt 对比 / 人在回路 |
| 核心机制 | keep/revert 棘轮 | 诊断→建议→验证 |
| 触发模式 | evals 分数下降时 | 对话暴露问题（用户纠正/效果不对）时 |

**触发时机**：用户使用 skill 执行任务时发现问题，或者 skill 调试开发时实测发现问题，主动触发园丁。

**被动原则**：没有用户指令，不主动扫描优化。

**例外**：在 Phase 5（测试验证）期间，园丁的被动原则**暂停**。此时园丁应主动监控对话质量，识别问题信号，在用户反馈之前主动介入。

## 架构设计：两层结构

园丁由两个核心部分构成：

```
┌─────────────────────────────────────────┐
│  底层优化部分                            │
│  · 诊断（Phase 0-1.5）                   │
│  · 建议生成（Phase 2）                   │
│  · 验证（Phase 5-5.5-6）                 │
│  · 循环优化（Phase 7）                   │
└─────────────────────────────────────────┘
                    ↓ 写入
┌─────────────────────────────────────────┐
│  skill画像库                            │
│  · 每skill一份画像                       │
│  · 每次优化的诊断+方案+验证结果           │
│  · 积累为「这类问题的解决方案」知识库     │
└─────────────────────────────────────────┘
```

**子agent职责边界**：

| Phase | 子agent职责 | 主agent职责 |
|---|---|---|
| Phase 1 | 分析对话，提取问题 | 协调子agent，监督分析质量 |
| Phase 1.5 | — | 主agent执行根本原因分析（这是判断，不是分析） |
| Phase 2 | 生成优化建议 | 审核建议，决定优先级 |
| Phase 3 | 设计测试场景和流程 | 审核测试设计 |
| Phase 5 | — | 监控对话，记录反应 |
| Phase 5.5 | — | 主agent执行成品扫描 |
| Phase 6 | — | 人类判断（用户） |

**子agent不可用时的fallback**：子agent不可用时，主agent承担所有任务，标注 `dry_run`。

---

**画像库数据结构**（每次优化记录）：

```
skill_name：[skill名]
时间：[优化时间]

【诊断】
问题：[具体描述]
根因：[一句话说明]

【优化操作】
改了什么：[具体改动]
为什么改：[对应诊断的哪条]

【验证】
结果：[用户反馈]
验证方式：dry_run / full_test
```

### skill画像库数据结构

每条优化记录包含：

```
skill_name：anna-russelmann-perspective
时间：2026-06-02

【诊断】
问题：成品草稿缺少世界观建立，冲突来得太突兀

【优化操作】

【验证】
结果：用户确认「这个版本更好一点」
```

**画像库的作用**：
- 积累优化经验，新skill进来时匹配类似案例
- 可追溯：为什么这样优化？有记录可查
- 不是模板，是项目需要什么存什么

### Phase 0: 接收问题

用户触发：「园丁处理一下」

**来源**：
- 用户使用 skill 执行任务时发现问题
- skill 调试开发时实测发现的问题

**接收**：
1. 确认目标 skill 是什么（用户指定，或从对话上下文中推断）
2. 确认问题是什么（用户描述 / 实测中发现）
3. 确认这是在优化哪个 skill 的哪个方向

**如果用户只说「园丁处理一下」，没有指定skill**：
- 从对话上下文中推断最近使用过的 skill
- 主动问：「要优化哪个 skill？」

```
项目目录结构：
```

### Phase 1: 对话分析（子 agent）

子 agent 专门分析对话记录，提取问题点。

**子 agent 任务**：

1. **回顾对话**：
   - 分析从 Phase 0 到当前的完整对话
   - 识别用户反复纠正、调整的地方
   - 识别「说到了但没完成」的地方

2. **提取问题**：
   - 什么是用户反复说要改的？
   - 什么是用户在对话中纠正过的？
   - 什么是对话暴露的设计缺陷？

3. **更新项目画像**：
   - 写入「对话分析.md」
   - 标记「关键纠正点」

**关键原则**：分析是为了提取「需要优化的地方」，不是总结对话内容。

---

### Phase 1.5: 追问根本原因

> **只运行于 Phase 1 之后。** 这是区分「修流程」和「改定位」的关键步骤。

对 Phase 1 提取的每个问题，追问一层：

**判断标准**：

| 问题类型 | 判断方法 | 解决方法 |
|---|---|---|
| **流程缺失** | 通过「增加环节/步骤」能解决吗？ | 在工作流里补步骤 |
| **能力不足** | 现有角色定位无法支撑这个需求吗？ | 在 skill 定位里改角色 |
| **方法不当** | 以上都不是？ | 建议换方向 |

**判断决策树**：

```
问题：{问题描述}
    ↓
增加一个步骤/环节，能解决这个问题吗？
    ↓ 能解决 → 「流程缺失」
    ↓ 不能解决 → 继续
        ↓
这个问题的原因，是 skill 的「角色定位」不够吗？
    ↓ 是 → 「能力不足」→ 建议改 skill 定位
    ↓ 否 → 「方法不当」→ 建议换方向
```

**判断标准详解**：

- **流程缺失**：问题出在「缺少一步」，加了就能解决
  - 信号：输出没有格式规范 → 加输出模板
  - 信号：没有用户确认点 → 加 Phase 4
  - 信号：成品草稿没人检查 → 加 Phase 5.5

- **能力不足**：问题出在「角色不够」，加步骤也解决不了
  - 信号：「Anna 不适合整部系列创作」（单集→整部，角色变了）
  - 信号：「用户想要AI输出成品，Anna 设计是用户输出」（定位冲突）
  - 信号：加了约束/步骤后问题依然存在

- **方法不当**：问题出在「方向/方法」本身
  - 信号：用错了创作框架
  - 信号：参考作品分析方向有误

**输出**：

每个问题输出：
```
问题：[问题描述]
类型：流程缺失 / 能力不足 / 方法不当
根因：[一句话说明]
建议方向：[描述]
```

**更新项目画像**：写入「对话分析.md」的「根本原因分析」章节

---

### Phase 2: 建议生成（子 agent）

基于 Phase 1 的分析，生成优化建议。

**子 agent 任务**：

1. **归类问题**：
   - 这个问题属于哪种优化类型？（P1-P5）
   - 根因是什么？（设计缺陷？场景不匹配？方法不对？）

2. **生成建议**：
   - 具体的优化方向
   - 每个方向的预期效果
   - 优先级排序

3. **更新项目画像**：
   - 写入「优化建议.md」
   - 每个建议说明「解决了什么问题」

---

### Phase 3: 设计测试（子 agent）

设计测试场景。

**子 agent 任务**：

1. **设计测试场景**：
   - 基于优化方向，设计能触发改动的测试 prompt
   - 确保测试场景真实（用户真正想写的题材）
   - 确保能触发「有改动的那个部分」

2. **设计测试流程**：
   - 人 + AI 一起对话的步骤
   - AI 观察记录的内容
   - 判断标准

3. **更新项目画像**：
   - 写入「测试设计.md」

---

### Phase 4: 等待确认

展示分析结果和优化建议，等用户确认：

**展示内容**：
- Phase 1 的问题分析汇总
- Phase 2 的优化建议方向（1-2个选项）
- 每个方向的预期效果

**用户决策**：
- 确认方向 → 进入 Phase 5
- 否定/修改 → 回到 Phase 1 重新分析
- 「不好」→「停」→「重新想」→ 回到 Phase 2

**Phase 5.5 成品扫描说明**：
- 如果 Anna 已输出成品草稿，Phase 4 前先执行 Phase 5.5
- Phase 5.5 发现问题 → 先问用户要不要改 → 用户确认后再进入 Phase 5 测试

---

### Phase 5: 测试验证

**园丁的测试（对话信号驱动），必须有人参与，不是 AI 跑对比。**

> **注意**：以下「思维类 skill」段落（line 508-525）是 Anna 实测的具体场景示例，**不是限定**——任何通过对话优化的 skill 都可以套用同样的测试逻辑。

---

#### 为什么不能只靠 AI 测试

- AI 没有真正的「创作灵感」 / 「业务体感」
- AI 的感受不是人的感受
- 人类判断的是「这个 skill 在真实场景里能不能解决问题」，不是「AI 跑出来的输出」
- 模板化的输出可能是 AI 没有发现更好的表达方式，不是 skill 的问题

---

#### 正确的测试方式

**人 + AI 一起执行测试验证**：

```
AI 发起对话
    ↓
人作为真实用户参与对话
    ↓
人在对话过程中实时感受效果
    ↓
AI 观察并记录人的真实反应
    ↓
对话结束后，AI 总结人的反馈
    ↓
判断：优化方向是否有效
```

---

#### 关键原则

1. **人必须真实参与对话**，不是看 AI 整理的输出
2. **AI 是观察者**，记录人在对话中的反应（犹豫/兴奋/困惑/选选项）
3. **关注对话过程**，不是关注输出结果
4. **测试场景要真实**：用用户真正遇到的问题场景，不是随意编一个 prompt

---

#### AI 监控探索阶段和执行阶段的边界（**Anna 案例参考**）

> **本节是 Anna 的具体场景**——其他 skill 应替换为该 skill 自己的「探索/执行」边界。

对话类 skill（Anna 类型）通常分两个阶段：
- **探索阶段**（定框架）：定故事方向、定核心角色、定世界设定
- **执行阶段**（写作）：具体写故事、文字输出

AI 在测试中要监控：
- 创作者是否跳到了「执行阶段」的思维（问「要教什么」「故事告诉我们什么」）
- 对话是否陷入了「清单式回答」而不是「探索式思考」
- 是否有「过早收拢」的现象（还没探索完就想要结论）

**发现问题时的介入原则**：

当创作者出现以下情况时，园丁要打断并拉回探索阶段：
- 问「这个故事能教小朋友什么」→ 拉回到「森林里发生了什么意外」
- 直接说「我要写一个关于XX的故事，目标是XX」→ 拉回到「为什么你想写这个」
- 跳过探索直接想要结论 → 拉回到「你脑子里现在有什么画面」

介入方式：用提问打断，不是给答案
- 「停，你先告诉我，这个故事里发生了什么意外？」
- 「在你说教育目标之前，我们先想清楚，森林里出了什么事？」

> **本节是 Anna 实测得出的具体场景**——其他 skill 测试时，应替换为该 skill 自己的「探索/执行」边界，不能照搬「故事探索」的措辞。

#### 对比测试（旧版本 vs 新版本）

如果需要对比两个版本：

```

    ↓
记录感受
    ↓
隔一段时间，再体验新版本
    ↓
对比两个版本的感受
    ↓
判断哪个版本更能解决问题（Anna 场景下是「创作体验」）
```

**关键**：两个版本要由同一个人体验，才能准确对比。

---

#### 什么情况下测试无效

- 人只是看了 AI 输出的对话记录，没有真正参与对话
- 测试 prompt 不是用户真正想写的题材
- AI 在对话中机械执行节点，而不是真正对话

### Phase 5.5: 成品扫描（主动检查）

**目的**：在 Anna 输出成品草稿后，园丁主动检查草稿质量，不等用户发现。

**什么时候执行**：
- Anna 完成第二阶段（输出完整剧集）后
- 在用户反馈之前
- 自动执行，不需要用户触发

**检查维度**：

| 维度 | 检查什么 | 问题信号 |
|---|---|---|
| 世界感 | 故事开头是否建立了世界和角色？ | 一来就是矛盾事件，太突兀 |
| 节奏 | 冲突是否来得太突兀？有没有铺垫？ | 用户说「太突然」 |
| 意外+成长 | 两者是否绑定？ | 只有意外没有成长 |
| 目标词分布 | 是否自然分布？ | 生硬植入、重复堆砌 |

**发现问题时的操作**：
1. 扫描发现维度有问题
2. **先问用户，不直接修改**
   > 「我注意到[维度]有问题：[具体描述]。要现在修改吗？」
3. 用户确认后 → 进入第三阶段（迭代优化）

**注意**：扫描是主动检查，发现问题后**问用户要不要改**，用户决定后再执行修改。

## 园丁 → 达尔文 交接协议（2026-06-03 新增）

> **核心问题**：之前园丁出方案后**没有移交达尔文**的工作流，导致方案停留在"对话信号"层，无法用 evals 量化验证。**本协议**定义园丁和达尔文之间的**交接格式 + 触发条件 + 失败回退**。

### 触发条件（什么时候移交）

> **分级原则**：**改动可逆性 × 量化必要性**。可逆 = 单 commit 可回滚；不可逆 = 改 skill 定位/触发词/边界声明。量化必要性 = evals 能兜底 vs evals 跑不出来。

| 园丁的判断 | 可逆性 | 移交达尔文？ | 理由 |
|-----------|--------|------------|------|
| 改 skill 定位 / 边界声明 / 触发词 | ❌ 不可逆 | ✅ **必须移交** | 影响范围大 + 不可逆，必须 evals 兜底 |
| 改 frontmatter description | ⚠️ 半可逆 | ✅ **必须移交** | 触发词改动影响加载频率，需要 evals 验证 |
| 加步骤 / 删步骤（流程缺失型） | ✅ 可逆 | ✅ **必须移交** | 流程改动需 evals 验证 |
| 加 pitfall / 改示例（细节型） | ✅ 可逆 | ⚠️ **可选移交** | 细节改动量化收益小，看用户时间预算 |
| Phase 1.5 判断是"方法不当"（换方向） | — | ❌ 不移交 | 方向问题由人决策，evals 跑不出方向对错 |
| 用户明确说"不需要 evals 验证" | — | ❌ 不移交 | 尊重用户意图 |
| skill 类型无法写 evals（创作类 / 对话类） | — | ⚠️ 可选移交 | 达尔文可能因无 evals 跑不了，需提前沟通；可选降级 dry_run |

**判断流程**：

```
园丁出方案
    ↓
【Q1】用户是否明确拒绝 evals？ → ❌ 跳过达尔文
    ↓ 否
【Q2】改动的可逆性？
    ↓ 不可逆（定位/边界/触发词）→ ✅ 必须移交
    ↓ 半可逆（frontmatter）→ ✅ 必须移交
    ↓ 可逆（流程/细节）
        ↓
        【Q3】改动的量化收益？
        ↓ 大（流程改动）→ ✅ 必须移交
        ↓ 小（细节）→ ⚠️ 可选移交（问用户时间预算）
```

### 园丁输出（移交前必须备齐）

**位置**：项目画像库的 `优化建议.md` 末尾新增"达尔文交接包"小节。

```
【达尔文交接包】

# 1. test-prompts.json
#    3-5 个测试 prompt，覆盖本次优化改动的所有维度
#    每个 prompt 含：id / prompt / expected_should_trigger / expected_reason

# 2. 改动摘要（≤200 字）
#    - 改了什么（具体段落/章节）
#    - 为什么改（对应诊断的哪条）
#    - 预期提升哪个维度（darwin 8 维 rubric 中的哪几条）

# 3. 验证目标
#    - 基线分数（如果有旧版本基线）
#    - 目标分数（优化后预期达到）
#    - 重点关注维度（哪 2-3 个维度变化最关键）

# 4. 耗时预估（**新增 2026-06-03**）
#    - 预估 evals 跑分耗时（分钟）：N 个 prompt × 估计单 prompt 耗时
#    - 是否需要 subagent 嵌套：会（≥5 次嵌套会触发 600s 超时）/ 不会
#    - 建议 eval_mode：full_test / dry_run
#    - **降级触发条件**：超过预估 2 倍耗时 → 自动降级 dry_run
#
#    示例：
#    {
#      "estimated_minutes": 8,
#      "needs_nested_subagent": false,
#      "suggested_eval_mode": "full_test",
#      "degrade_to_dry_run_after_minutes": 16
#    }

# 5. 路径前置检查（**新增 2026-06-03**）
#    园丁在交接前**自己跑一遍**这些检查——避免达尔文接收时才发现路径错：
#    [ ] 目标 skill 目录存在：`~/.hermes/skills/<skill>/` 或 profile 专用路径
#    [ ] 目标 SKILL.md 可写（无只读 / 无冲突未提交）
#    [ ] test-prompts.json 格式合规（参考 darwin-skill/test-prompts.json 模板）
#    [ ] 8 维 rubric 适用（创作类 / 元组件类需提前标注）
#    [ ] 上一次达尔文跑分记录存在（results.tsv）—— 没有则需要先跑基线
```

**为什么要加路径前置检查**：达尔文不是 hermes 子 agent，是个**独立可加载的 skill**。如果目标 skill 目录不存在 / SKILL.md 只读 / 之前没跑过达尔文，**园丁交接后达尔文第一步就失败**——浪费整个流程。园丁提前跑这些检查是**交接前的健康检查**。

### 达尔文接收（拿到交接包后做什么）

1. **加载 test-prompts.json** → 写到自己 skill 目录的 test-prompts.json
2. **跑 8 维 rubric 基线**（如果交接包给了基线分数则跳过）
3. **应用园丁的改动建议**（写入 SKILL.md）
4. **跑 8 维 rubric 优化后** → 对比基线
5. **棘轮决策**：
   - 新分数 > 旧分数 → **keep**，git commit "darwin: 验证园丁方案 (commit hash)"
   - 新分数 ≤ 旧分数 → **revert**，git revert，**回退通知园丁**重新设计

### 失败回退（达尔文验证不通过怎么办）

| 失败原因 | 园丁怎么办 |
|---------|-----------|
| 改动导致 evals 分数下降 | **回退**改动 → 园丁重新分析（回到 Phase 1） |
| test-prompts 触发不到改动点 | **改 test-prompts** → 园丁补 prompt |
| 8 维 rubric 不适用（创作类 skill） | **跳过达尔文** → 直接 Phase 6 人类判断 |
| 达尔文跑分超时 / 子 agent 不可用 | **降级 dry_run** → 标注 `eval_mode: dry_run` → 不阻塞园丁流程 |

### 联动示例（完整路径）

```
用户在用 picturebook-creator 时发现目标词分布不自然
    ↓
【园丁】
Phase 1: 读对话，提取问题"目标词重复堆砌"
Phase 1.5: 判断"流程缺失"（缺目标词分布检查步骤）→ 可逆 + 量化收益大 → 必移交
Phase 2: 出方案——加"目标词自然分布检查"到第二阶段
Phase 3-5: 测试场景设计 + 人在回路验证
    ↓ 方案通过
Phase 9.5: 移交达尔文
    ├── 写 test-prompts.json（5 个 prompt，覆盖目标词分布场景）
    ├── 改动摘要：picturebook-creator 第二阶段新增"目标词自然分布检查"步骤
    ├── 验证目标：8 维 rubric 维度 8（实测表现）从 6 → 8
    ├── 耗时预估：5 prompt × 估计 1.5 分钟/prompt = 7.5 分钟，eval_mode=full_test
    └── 路径前置检查：~/.hermes/profiles/huiben/skills/creative/picturebook-creator/ 存在 ✓
    ↓
【达尔文】
加载 test-prompts → 跑基线（6）→ 应用改动 → 跑优化后（8）→ keep
    ↓
【园丁】
Phase 9: 更新画像库"验证记录.md"——达尔文确认有效（**顺序：9.5 → 9**）
```

### Pitfall: 园丁不能跳过达尔文直接给最终方案

**触发条件**：园丁 Phase 5 测试通过后，直接进入 Phase 6 人类判断 + Phase 7 迭代 + Phase 8 提交，**没经过达尔文量化验证**。

**错误后果**：
- 方案停留在"对话信号"层，无法用 evals 兜底
- 改动可能引入新 bug，但人眼难以识别（量化指标才能发现）
- 4-skill 工具包联动机制空转

**正确做法**：
- **方案通过 Phase 5 后必须走 Phase 9.5 移交达尔文**
- **达尔文 keep 才能进 Phase 6 人类最终判断**
- **达尔文 revert 必须回到 Phase 1 重新分析**

## 与达尔文的关系

园丁和达尔文是 **4-skill 工具包的联动关系**，不是简单的"互补"——**园丁出方案，达尔文做量化验证**。完整联动协议见上文"园丁 → 达尔文 交接协议"段。

### 决策矩阵（一图看懂）

| skill 哪里出问题 | 工具 | 原因 |
|----------------|------|------|
| evals 分数下降 / 多 prompt 对比 | **达尔文**（量化评分 + 棘轮） | 有量化数据，按分数走 |
| 真实对话暴露问题（用户反馈、效果不对） | **园丁**（读对话 + 给方案） | 没有量化数据，靠人观察 |
| 园丁方案需要量化验证 | **达尔文**（接 Phase 9.5 移交） | 4-skill 工具包联动 |
| 整体 skill 库整理 / 审计 / 归档 | **skill-organizer** | 全 skill 集合视角 |
| 写新 skill / 改 frontmatter | **skill-creator**（主管道） | 编写视角 |

### 本质差异

- **达尔文** = 「**多 prompt 跑分对比**」—— 适合能写 evals 的 skill（操作流程型、有明确测试点）
- **园丁** = 「**单次对话深度分析**」—— 适合不能用 evals 量化的 skill（创作类、对话类、需要「人在回路」的 skill）

### 互补用例（2026-06-03 更新）

- Anna（对话驱动型） → 园丁主，达尔文辅（如果能写 evals）
- hermes-gateway-result-routing（操作流程型） → 达尔文主，园丁不介入
- skill-creator 本身的优化 → skill-creator 自带 evals + darwin 量化（不归园丁）
- **任何 skill 的优化** → 园丁出方案 → 达尔文验证（**新机制，2026-06-03**）

## 设计原则

1. **被动接收**：没有用户指令，不主动扫描优化
2. **固定方法**：提取方法（收集→归类→验证→沉淀）是标准步骤，不是可选工具
3. **项目画像**：每个优化任务独立项目模块，阶段性成果实时保存
4. **子 agent 架构**：Phase 1/2/3 由子agent执行，Phase 1.5/5.5/6 由主agent/用户执行（详见职责边界表）
5. **人在回路**：所有判断由人类完成，园丁只提供信息和选项
6. **可回滚**：所有改动在分支上，人类判断前不修改 dev/master
7. **版本可见**：所有分支推送到 GitHub，用户可以查看 diff
8. **循环优化**：不满意就继续迭代，直到用户满意

## 禁止的优化方向（任何版本都必须拦截）

| 禁止 | 原因 |
|---|---|
| 把对话诊断变成「清单式输出」 | 园丁的输出应该是「从对话里发现什么 + 建议改什么」，不是「10 步操作手册」 |
| 只给格式不改原理 | 园丁出方案时必须说「为什么这样改」——只说「加这一节」没价值 |
| 把「对话信号」当 evals 数据 | 园丁看的是真实对话，不能拿多 prompt 跑分当输入 |
| 跳过「对话分析」直接给方案 | 没分析过的优化建议 = 拍脑袋，违反园丁的「诊断→建议」核心机制 |
| 把园丁当达尔文用 | 园丁不做 evals 跑分（那是达尔文）；达尔文也不读对话（那是园丁） |

### Pitfall: 修改 skill 仓库前必做 git 同步检查——本地 dev 跟 origin 不一致会丢别人的 commit

**触发条件**：修改一个独立 git 仓库的 skill（`~/.hermes/skills/<skill>/` 是独立 repo）然后 commit，**不**先 fetch + 检查 origin 跟本地 HEAD 是否一致。

**错误后果**（2026-06-01 darwin-skill 实测）：
- 本地 dev HEAD = `2056abf`（指向 master 内容）
- origin/dev HEAD = `c6ac79f`（有 143 行"思维方式型skill保护模式"）
- **没察觉 diverged**——在本地改了 7 行 frontmatter 直接 commit
- diff 显示 `-143 / +41` = **静默丢了别人的 143 行**
- 立即 reset + reset --hard origin/dev 才恢复
- 重新 patch +7 行 → 干净 commit

**正确做法**（**所有 skill 仓库 commit 前必跑**）：
```bash
# 1. 拉最新
git fetch origin <branch>

# 2. 验证一致性
git rev-parse HEAD               # 本地
git rev-parse origin/<branch>    # 远程
# 必须**完全一致**才继续

# 3. 不一致时：reset --hard 同步（**别 rebase 别人的 commit**）
git reset --hard origin/<branch>
# 重新 patch 你的改动

# 4. 验证 diff 范围
git diff --stat
# **预期**：改动量是预期的一半以内
# 超过 10 倍 = 立即 git reset + 重新做
```

**判断标准**：
- `git diff --stat` 改动量 **≤ 预期** → 安全
- 改动量 **= 预期 × 2-5** → 检查是不是合并了别人的内容
- 改动量 **> 预期 × 10** → 立即 `git reset --hard origin/<branch>` + 重新做

**预防配置**（强烈建议每个 skill 仓库加）：
```bash
# pre-commit hook（hooks/pre-commit）
#!/bin/bash
EXPECTED_LINES=20  # 每次改的预期行数
ACTUAL=$(git diff --cached --stat | tail -1 | grep -oP '\d+' | head -1)
if [ "$ACTUAL" -gt $((EXPECTED_LINES * 10)) ]; then
  echo "ERROR: diff 改动量 $ACTUAL 超过预期 $EXPECTED_LINES × 10"
  echo "可能本地 HEAD 跟 origin 不一致，先 git fetch + reset --hard origin/<branch>"
  exit 1
fi
```

**关联**：完整 git 同步 + 安装流程 + push 网络 workaround 见 `hermes-multi-agent/references/skill-installation-procedure.md`。

### Pitfall: patch 工具偶发"删除未指定段落"——SKILL.md 长文件必跑 diff 范围检查

**触发条件**：用 `patch` 工具修改**长** SKILL.md（>1000 行）时，**old_string 只匹配标题但实际删除范围超出预期**。

**错误后果**（2026-06-03 picturebook-video v10 升级实测）：
- 操作：给 v3 范式正文加"❌ 已弃用"标记（4 行改动）
- 实际 diff：`+4 / -42`（删了 42 行 = v9 出现的工作方法论教训 + 单测门 SOP 段）
- 这些**不属于 patch 范围**——但被 patch 工具静默删了
- 恢复：`git checkout HEAD -- SKILL.md` → 重新 patch（更小心 + 更小 old_string）→ 验证 `+11/-5` 合理
- 教训写入 commit message（`bea1645`）

**根因推测**：patch 工具对 unicode 行号定位 + 上下文匹配有边界 case——长文件中**同标题多次出现**或**old_string 模糊匹配**可能误删。

**正确做法**（**所有 SKILL.md 修改后必跑 3 步验证**）：

```bash
# 1. diff 范围合理性
git diff --stat
# 预期：改动量 ≤ 预期 × 1.5
# 超出 = 立即 git checkout HEAD -- <file> + 重新做

# 2. 关键内容还在
grep -c "<关键段标题1>\|<关键段标题2>" SKILL.md
# 验证：count 应该跟修改前一致

# 3. patch 范围可控
# 永远用**最小** old_string（单行标题 + 1 行上下文）而非大段
```

**判断标准**：
- diff 改动量 **≤ 预期** → 安全
- 改动量 **= 预期 × 2-5** → 立即 `git diff` 细看是不是误删
- 改动量 **> 预期 × 5** → 立即 `git checkout HEAD -- <file>` + 重新做

**预防**：
- patch 时 `old_string` **尽量只含 1 行**（标题行）+ 必要的最小上下文
- 长 SKILL.md (>1500 行) 优先**用 write_file 全量重写**而非 patch
- patch 工具 + 写完 grep 关键内容**必跑**——不要信任 patch 工具的"我只改了 X 行"

**检测方法**：当 `git diff --stat` 改动量跟"你写了几行"对不上时，**怀疑误删**。

- 触发词清晰区分：园丁触发 = "对话诊断/方案（单 skill）"；skill-organizer 触发 = "整理/审计/分类（多 skill）"

**检测方法**：用户提到"全局"或"多个 skill"时，**先考虑 skill-organizer**，不要默认"我管"。
