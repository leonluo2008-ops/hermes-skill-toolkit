# Gardener Skill 知识沉淀

## 背景

2026-06-01 发现：达尔文的棘轮机制只适用于「可量化」的 skill 优化。思维方式型 skill（Anna、Spike Jonze）的核心价值是「启发性」「创作空间」，这些指标无法量化，所以：

- Darwin 的打分机制失效
- 思维方式型 skill 被 Darwin 优化后会变差（开放性被压缩）
- 需要独立工具「园丁」来处理这类 skill

## 路由规则

```
达尔文遇到 Skill → 先判断类型
    ↓
是思维方式型吗？ → 是 → 路由到园丁
    ↓ 否
继续标准达尔文流程
```

## 关键决策：Anna 重构

Anna skill 的教训：
- 最初版本有「第1步→第2步→第3步」结构
- Darwin 保护模式识别时产生分歧（结构 vs 内容特征）
- 解决方式：彻底移除五步架子，改为「节点一/节点二/节点三」纯节点结构
- 删除了「写完后检查」检查清单（这是明确禁止的）

重构后 commit: `dcc9c5d`

## 园丁分支命名规范

```
gardener/YYYYMMDD-优化摘要
示例：gardener/20260601-反例补充
```

## 人类判断标准（Phase 4）

四个维度，判断「创作空间」是否被压缩：
1. 创作者能形成自己的画面吗？
2. 有没有套路感？
3. 开放性有没有被压缩？
4. 整体启发性变好了吗？

决策：偏好哪个版本 → 哪个保留

## 关键教训：头皮方向事件（2026-06-01）

**错误做法**：
```
用户说加头皮方向 → 直接加 → 测试
```
跳过了提取方法的验证步骤。

**正确做法**：
```
1. 收集：头皮方向从哪来？（用户反馈，说明用户在实际创作中有头发场景需求）
2. 归类：属于 P2「如果」问题库扩展
3. 验证：是否符合 Anna 原则？（物理因果链 ✓ 小社会 ✓ 启发性 ✓）
4. 沉淀：写入节点二「如果」问题库
```

**教训**：用户说「加什么」是信号，不是结论。提取方法必须走完四步，不能跳过验证直接写入。

## 对比测试 prompt 设计原则

- 测哪个维度 → 用哪个维度的 prompt
- 用「眼睛」prompt 测「头皮」方向 → 两版输出相同，测不出差别
- 不是 feature 无效，是 case 选错了维度
