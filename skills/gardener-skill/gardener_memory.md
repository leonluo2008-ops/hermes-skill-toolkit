# gardener_memory.md — 跨次诊断策略记忆

> **位置**: `gardener-skill/gardener_memory.md` (本文件, 模板态)
> **用途**: 跨多次 gardener 诊断沉淀的**通用教训**, 避免重复犯同样的错
> **借鉴**: microsoft/SkillOpt `meta_skill.md` 的形式 (简短 / 结构化 / 按 epoch 滚动)

---

## 写入规则 (硬约束)

**什么时候写**:
- 每次 gardener Phase 5.5 (成品扫描) 完成时
- Phase 6 人类判断通过后, 落地到本文件
- **绝不**在 Phase 1-5 中途随手写

**写多少**:
- 每次诊断 1-3 条 (不是越多越好)
- 每条 1 行, 可执行 (动词开头)
- 沉淀**通用规律**, 不是**具体案例**

**容量管理**:
- 上限: 最近 **10** 条
- 超过 10 条: 自动淘汰最早的 (按写入时间倒序)
- 淘汰前 1 个月内的条目需要**二次确认** (避免误删黄金教训)

---

## 沉淀标准 (反"绑定具体内容"铁律)

**该写** (通用方法):
- ✅ "需要外部 API 的 skill 在 description 里必须明确触发词 + 何时用"
- ✅ "多文件改动的 skill, evals 必须覆盖跨文件场景"
- ✅ "触发词跟 superpowers / skill-creator 重叠时, 在 description 里加否定条件"

**不该写** (绑定具体内容, 违反铁律):
- ❌ "yoto rule checker 缺规则 3" → 具体 skill 的具体问题
- ❌ "anna-russelmann-perspective 头皮方向错了" → 具体内容
- ❌ "2026-06-XX 的某个对话里 X 用户说 Y" → 案例不是教训

**判断标准**: 把这条教训放在**另一个 skill** 上, 还成立吗?
- 成立 → ✅ 写入
- 不成立 → ❌ 删掉, 那是 case-specific

---

## 读取时机

- 每次 gardener Phase 1 (对话分析) 开始前, 读取本文件**头部 5 条**
- 头部 5 条作为"已知教训"叠加进分析上下文
- **不**全部读 (避免淹没对话信号本身)

---

## 写入格式 (严格)

```markdown
## YYYY-MM-DD | <skill name 或 "通用">

- 教训 1: <1 行, 动词开头, 可执行>
- 教训 2: <1 行> (可选)
- 教训 3: <1 行> (可选)
- 适用场景: <一类 skill 关键词, 用于检索时匹配>
```

**示例** (空模板态保留格式, 不放实际条目):

```markdown
## 2026-06-XX | gardener-template

- 教训 1: <待首次写入>
- 适用场景: <待定>
```
