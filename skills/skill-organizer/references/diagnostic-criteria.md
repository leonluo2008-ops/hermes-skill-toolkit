# Skill Organizer 诊断标准参考

> 本文档为 skill-organizer 提供类型预分 + 5维度诊断评分的详细 rubric，供实际操作时参照。
> 新增「维度六：类型预分」作为前置过滤，不记分，只决定是否参与后续评分。

---

## 维度六：类型预分（前置过滤，不记分）

### 分类标准

```
【通用工具型】→ 直接免检，不参与归档评分
【业务专用型】→ 进入 5 维度评分
```

### 判定规则

满足任一即属于**通用工具型**：

| 类别 | 典型 skill | 说明 |
|------|-----------|------|
| 平台集成（飞书套件） | lark-* | 日历、文档、即时通讯、审批、任务等 |
| 平台集成（其他） | notion、obsidian | 知识管理、笔记 |
| GitHub 工具 | github-* | PR、issues、repo 管理 |
| MCP 工具 | mcporter、native-mcp | MCP 协议相关 |
| 调试工具 | systematic-debugging、python-debugpy、node-inspect-debugger | 调试相关 |
| Skill 封装 | skill-organizer、hermes-agent-skill-authoring、find-skills | skill 开发封装 |
| Agent 开发 | claude-code、codex、opencode、hermes-multi-agent | 多 Agent 开发 |
| 通信/协作 | lark-im、lark-task、lark-calendar、lark-contact | 即时通讯、日历、任务 |
| 文档/表格 | docx、xlsx、powerpoint、lark-sheets | 文档处理 |
| 开发环境 | jupyter-live-kernel、python-debugpy、node-inspect-debugger | 开发环境 |

**判定原则：**
- 跨业务、跨行业均可使用的 → 通用工具型
- 行业垂直应用（如绘本、抖音运营、AI视频、漫剧创作）→ 业务专用型
- Agent 开发框架（hermes-multi-agent、claude-code 等）→ 通用工具型（因为是开发工具，不是业务本身）

**⚠️ 常见误判警告（必须校验）：**
以下 skill 常被错误地放入 universal_tools，实际应为业务专用型：
- `picturebook-creator` → 绘本创作，核心业务，应进入诊断评分
- `picturebook-video` → 绘本转视频，核心业务，应进入诊断评分
- `douyin-ops` → 抖音运营，核心业务，应进入诊断评分
- `ai-drama` → 漫剧创作，业务相关，应进入诊断评分
- `ai-drama-storyboard` → 漫剧分镜，业务相关，应进入诊断评分
- `seedance2.0-tool` → 视频生成模型调用，垂直业务工具，应进入诊断评分

**判定顺序：**
1. 先查 `data/user-universal-tools.yaml` 中的用户自定义免检列表
2. 检查列表准确性——若列表中出现了上方的「误判警告」中的 skill，先将其移出 universal_tools，重新归类为业务专用型
3. 再用上述类别规则做兜底判断
4. 落入任一类别即标记为通用工具，不进入评分

### ⚠️ 多Agent架构：跨Agent的skill应从当前Agent池子里物理移除（2026-05-29修订原则）

**核心原则：用不上的skill，不应该存放在agent的skill清单里。**

在多Agent系统中，每个Agent的skill pool应该只包含它自己会用到的skill。
- 调度型/中枢型Agent的SOUL.md设计为**路由转发**，不自己执行内容创作
- 内容创作类skill（picturebook-creator、douyin-ops、ai-drama等）**物理上**应该只在执行型Agent（huiben）的profile里
- 主Agent的全局skills池（`~/.hermes/skills/`）里有这些skill的副本 → **矛盾**：物理上能调用，但SOUL说不要调用

**正确做法：**
- 诊断时发现当前Agent的池子里有属于其他Agent的skill → **建议物理移除**
- 不是说「在诊断报告中标记为排除，然后保留文件」
- 而是说「这个skill不应该在本Agent池子里，建议移走/删除」

**判断规则：**

| 情况 | 正确处理 |
|------|---------|
| 当前Agent是调度型，skill属于执行型Agent | **建议从本Agent池子物理移除**，在诊断报告中标记为「归属：huiben → 建议移除」 |
| skill属于huiben的内容创作范围 | 诊断主Agent时：发现 → 建议移除；诊断huiben时：该skill在huiben自己的池子里，正常诊断 |
| 全局池里有副本但huiben也有 | 全局池副本 → 建议移除；huiben专属版本 → 保留在huiben |

**操作方式：**
- 诊断主Agent时：发现跨Agent skill → 标记为「归属huiben，建议移除（全局池副本）」，不进入业务专用型诊断
- 在诊断报告的「本次建议移除」列表中列出这些skill
- 用户确认后，执行物理删除（skill_manage delete）

**总结：** 调度型Agent的skill-organizer只需要诊断「通用工具」和「自身业务专用型」。不属于本Agent的skill（无论是转发给其他Agent的还是当前Agent用不上的），都应该在诊断报告中建议物理移除，而不是「排除后保留文件」。

### 通用工具型 skill 的处理规则

```
在 universal_tools 列表中 → 标记为「✅ 通用工具（免检）」
  → 全程跳过诊断评分
  → 不进入归档候选
  → 诊断报告中独立展示

不在列表中 → 进入 5 维度评分
```

## 维度一：角色定位匹配度（权重 40%）

### 评分标准

| 得分 | 场景 | 说明 |
|------|------|------|
| 9-10 | skill 核心能力与用户定位完全吻合 | 属于核心职责，如"绘本创作者"使用 picturebook-creator |
| 7-8 | skill 能力在定位范围内，但非核心 | 如日常助手中偶尔使用的飞书技能 |
| 5-6 | skill 与定位有交集但模糊 | 需要进一步判断使用频率 |
| 3-4 | skill 与定位关联度低 | 如游戏 skill 用于工作导向用户 |
| 1-2 | skill 与定位完全不相关 | 如用程序员工具处理创意工作 |

### 判断依据

- 读取 skill 的 `description` 和 `triggers` 字段
- 对比用户角色定位（从 memory/USER.md 或手动输入获取）
- 关注 skill 的 `category` 和 `tags`

---

## 维度二：使用频率（权重 25%）

### 评分标准

| 得分 | 场景 | 说明 |
|------|------|------|
| 9-10 | 每周使用 ≥3 次 | 属于日常高频工具 |
| 7-8 | 每周使用 1-2 次 | 属于常规工具 |
| 5-6 | 每月使用 2-3 次 | 属于低频工具 |
| 3-4 | 每季度使用 1-2 次 | 属于偶用工具 |
| 1-2 | 半年以上未使用 | 可能需要归档 |

### 判断依据

- 参考 Hermes Agent 的 `.usage.json` 数据（若可用）
- 询问用户确认
- 如无数据，按定位匹配度推断（高匹配通常也高频率）

---

## 维度三：冲突风险（权重 15%）

### 评分标准

| 得分 | 场景 | 说明 |
|------|------|------|
| 9-10 | 与其他 skill 完全不重叠 | 功能互补，无冲突 |
| 7-8 | 有少量重叠但可区分 | 如两个聊天工具但场景不同 |
| 5-6 | 功能重叠需用户选择 | 如两个笔记 skill |
| 3-4 | 高度重叠 | 功能重复，应合并或归档其一 |
| 1-2 | 完全重复 | 其中一个应删除 |

### 判断依据

- 列出所有 skill 的核心功能
- 两两比较重叠度
- 优先保留与定位更匹配的那个

---

## 维度四：维护成本（权重 10%）

### 评分标准

| 得分 | 场景 | 说明 |
|------|------|------|
| 9-10 | 无外部依赖，即插即用 | 不需要 API 密钥或特殊配置 |
| 7-8 | 有配置但稳定 | 如飞书 token 等已配置好的凭证 |
| 5-6 | 有过期风险但可控 | 如 API 限额较低但用户不常用 |
| 3-4 | 经常需要维护 | 如第三方 API 变更频繁 |
| 1-2 | 几乎不可用 | 如依赖已下线服务 |

### 判断依据

- 检查 skill 的 `required_environment_variables` 和 `required_commands`
- 检查 skill 的 `missing_required_environment_variables` 和 `missing_credential_files`
- 询问用户是否有维护负担

---

## 维度五：可替代性（权重 10%）

### 评分标准

| 得分 | 场景 | 说明 |
|------|------|------|
| 9-10 | 不可替代 | 核心能力无其他 skill 可覆盖 |
| 7-8 | 难以替代 | 替代成本高，用户会不方便 |
| 5-6 | 可以替代 | 有其他 skill 可以覆盖大部分功能 |
| 3-4 | 容易替代 | 其他 skill 已可完全覆盖 |
| 1-2 | 完全可替代 | 功能已被其他 skill 包含 |

### 判断依据

- 列出该 skill 的核心功能
- 检查其他 skill 是否提供相同功能
- 考虑功能整合的可行性和用户接受度

---

## 综合评分计算

```
总分 = (维度一得分 × 0.4 + 维度二得分 × 0.25 + 维度三得分 × 0.15 + 维度四得分 × 0.1 + 维度五得分 × 0.1) × 10
```

### 分类阈值

| 分类 | 分数区间 | 行动 |
|------|----------|------|
| **保留** | ≥70 分 | 与当前定位高度匹配，建议保留 |
| **待定** | 30-69 分 | 可能有用但定位模糊或功能重叠，需用户确认 |
| **归档** | <30 分 | 定位不匹配/长期未用/功能重复，建议移出 |

> ⚠️ **归档阈值已调整**：从 <40 调整为 <30，避免一刀切，给「待定」更多空间。

## 快速决策树

```
用户说"整理skill"时：
  1. 读取 skill 清单（skills_list）
  2. 检查 data/user-universal-tools.yaml 是否存在
     → 无：触发初始化流程（读画像→生成候选→用户确认→写入）
  3. 获取用户定位（memory/USER.md 或手动询问）
  4. 类型预分：查 universal_tools 列表 + 类别兜底规则
     → 通用工具型：标记免检，跳过诊断
     → 业务专用型：进入步骤 5
  5. 对业务专用型 skill 打分（5 维度）
  6. 汇总分数，分类为 保留/待定/归档
  7. 展示诊断报告给用户
  8. 等待用户确认「待定」列表的归属
  9. 执行归档/删除操作
  10. 输出整理报告
```

## 常见场景示例

### 场景 A：绘本创作者的 lark-calendar
- 类型预分：通用工具型（飞书套件）→ **直接免检，不评分**

### 场景 B：绘本创作者的 picturebook-creator
- 维度一（匹配度）：10 分（核心职责）
- 维度二（频率）：8 分（每周使用）
- 维度三（冲突）：9 分（无重叠）
- 维度四（成本）：8 分（配置稳定）
- 维度五（可替代）：9 分（不可替代）
- **综合**：10×0.4 + 8×0.25 + 9×0.15 + 8×0.1 + 9×0.1 = 4+2+1.35+0.8+0.9 = 9.05 → **保留**

### 场景 C：有 Notion 使用场景的 notion
- 类型预分：通用工具型（知识管理平台）→ **直接免检，不评分**

### 场景 D：绘本创作者的 minecraft-modpack-server
- 类型预分：业务专用型
- 维度一（匹配度）：2 分（与创意定位无关）
- 维度二（频率）：2 分（长期未用）
- 维度三（冲突）：8 分（无重叠）
- 维度四（成本）：9 分（无依赖）
- 维度五（可替代）：9 分（无替代需求）
- **综合**：2×0.4 + 2×0.25 + 8×0.15 + 9×0.1 + 9×0.1 = 0.8+0.5+1.2+0.9+0.9 = 4.3 → **归档**（<30分）

### 场景 E：GitHub 重度用户的 github-pr-workflow
- 类型预分：通用工具型（GitHub 工具）→ **直接免检，不评分**
